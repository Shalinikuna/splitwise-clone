## 3 Cases Where AI Produced Wrong Output & How I Caught and Fixed It

### Case 1 — Incorrect `unique_together` vs `UniqueConstraint`

**What AI generated:**
```python
class Meta:
    unique_together = ("group", "user")
```

**Why it was wrong:**
Django deprecated `unique_together` in favour of `UniqueConstraint` in Django 4.x. The migration generated by AI used the old style, which caused a warning during `makemigrations` and would fail on newer versions.

**How I caught it:**
Running `python manage.py makemigrations` showed a system check warning: `WARNINGS: api.groups.GroupMember: (models.W036) 'unique_together' is deprecated...`

**What I changed:**
```python
class Meta:
    constraints = [
        models.UniqueConstraint(fields=["group", "user"], name="unique_group_user")
    ]
```
Updated both the model and the migration file manually.

### Case 2 — WebSocket Auth Middleware Used Wrong Scope Key

**What AI generated:**
```python
class JWTAuthMiddleware(BaseMiddleware):
    async def __call__(self, scope, receive, send):
        token = scope["headers"].get(b"authorization", b"").decode()
        # ... parse token
```

**Why it was wrong:**
WebSocket connections in browsers **cannot** set custom HTTP headers. The `Authorization` header is not sent on WebSocket upgrade requests from browser JavaScript. This meant every WebSocket connection would silently fail authentication and immediately close.

**How I caught it:**
When testing the chat feature with two browser tabs, the WebSocket console showed `code: 4001` (closed by server) immediately after connection. Adding `console.log(scope["headers"])` in the consumer showed the authorization header was missing entirely.

**What I changed:**
Switched to passing the JWT as a URL query parameter `?token=<jwt>` and reading it from `scope["query_string"]`:
```python
query_string = scope.get("query_string", b"").decode()
params = parse_qs(query_string)
token = params.get("token", [None])[0]
```
Frontend updated to: `new WebSocket(\`ws://.../ws/expenses/${id}/?token=${localStorage.getItem('access')}\`)`

### Case 3 — CSV Percentage Normalisation Applied Before Validation

**What AI generated (first version of engine.py):**
```python
# AI normalised percentages immediately when parsing
for k in parsed_splits:
    parsed_splits[k] = parsed_splits[k] / total_pct * 100
# Then logged warning AFTER modifying data
```

**Why it was wrong:**
The anomaly log showed the normalised values (which sum to 100%) instead of the original erroneous values (which summed to 110%). This meant the report was misleading — it looked like the percentages were fine because it showed the corrected numbers, not the raw problem.

**How I caught it:**
When reviewing the anomaly report output for Row 29, the message said "percentages sum to 100%" (because the data had already been normalised before the check ran). The original CSV clearly showed `30+30+30+20=110%`.

**What I changed:**
Reordered the logic — validate first, log the anomaly with original values, then normalise:
```python
total_pct = sum(parsed_splits.values())  # calculate from original data
if abs(total_pct - 100) > 0.5:
    add_anomaly(..., f"percentages sum to {total_pct}% – normalised proportionally")
# THEN normalise
for k in parsed_splits:
    parsed_splits[k] = (parsed_splits[k] / total_pct * 100).quantize(...)
```
The anomaly message now correctly shows the original wrong total before correction.
